## Findings considered and rejected

- **Convert Devin to a generic custom ACP provider (Paseo-style catalog)** —
  rejected by ADR 0001; Devin stays a first-class `ProviderKind`. Do not re-litigate.
- **Implement rollback/revert for Devin now** — deliberately unsupported this
  release; `rollbackThread` keeps its explicit `ProviderAdapterRequestError`
  (DevinAdapter.ts:810-826) until native Devin revert/fork semantics are mapped
  (locked decision 4). A follow-up may map discovered `/revert`//`/fork` slash
  commands (see Plan 004 maintenance notes).
- **Create a `DevinCommandCatalog` static fallback** — locked decision says
  allowlist only if full discovery proves impossible; Plan 004 implements full
  discovery and returns an empty list otherwise, so no catalog module is created.
- **Thread compaction support** — likely unsupported by Devin ACP; capability
  stays `supportsThreadCompaction: false` until proven otherwise.
- **URL/browser-based elicitation and auth** — out of scope per locked decision 11;
  Plan 003 explicitly declines `mode: "url"` elicitations.
- **Removing `MODEL_OPTIONS_BY_PROVIDER.devin` from contracts** — it stays as the
  single static list; Plan 001 makes the server catalog derive from it instead of
  duplicating it.
- **Treating raw `user-input.resolved` answer persistence as Devin-only** — rejected
  during final review because Cursor, Claude, Codex, and OpenCode already emit raw
  answers too. Plan 009 handles this as a cross-provider ingestion policy.
